Clean code is not about style guides or linters — it is about writing code that the next person (often future you) can read without frustration. A few rules cover most of the ground.

1. Name things clearly

A well-chosen name eliminates the need for a comment. Compare:

// bad
const d = new Date(ts * 1000)

// good
const publishedAt = new Date(timestampSeconds * 1000)

If you find yourself adding a comment to explain a variable, rename it instead.

2. Keep functions small

A function should do one thing. A practical test: if describing what it does requires the word "and", split it. Functions that fit on a screen are easier to test, easier to move, and easier to delete.

3. Delete unused code

Dead code is a reader tax. Every commented-out block, every unused import, every feature flag that was never cleaned up — all of it costs future readers time. Version control has the history; you do not need to keep it in the source.

4. Make the happy path obvious

Handle edge cases early and return, so the main logic is not buried in nested conditions:

// harder to follow
function process(item) {
    if (item) {
        if (item.active) {
            // ... main logic deeply nested
        }
    }
}

// cleaner
function process(item) {
    if (!item) return
    if (!item.active) return
    // main logic at the top level
}

5. Consistency beats perfection

A codebase where every file follows the same conventions is easier to navigate than one where each file reflects a different author's preferences. Pick an approach and apply it everywhere — even if it is not the optimal one.

The best code is the code your teammates can maintain without asking you questions.
