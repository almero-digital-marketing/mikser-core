Getting Started with Mikser

January 10, 2026
introbasics

Mikser is a static site generator built around a simple idea: your content lives in documents, your templates live in layouts, and a small config wires them together. This post walks through the basics to get you up and running.

Project structure

A typical Mikser project looks like this:

my-site/
├── documents/        # your content (Markdown, HTML, YAML, …)
├── layouts/          # Handlebars templates
├── resources/        # referenced assets (images, videos)
├── mikser.config.js  # plugin list & options
└── out/              # generated output

Run mikser and the engine imports all documents, matches them to layouts, renders the HTML, and writes the output folder.

Defining a layout

Create a file in layouts/ with a double extension. The first extension sets the output format, the second sets the template engine:

  • post.html.hbs — renders HTML with Handlebars
  • feed.xml.hbs — renders XML with Handlebars
  • report.html-pdf.hbs — renders HTML and then converts it to PDF

Inside the layout, the current document is exposed as document. Metadata from front-matter lives under document.meta and the body content is in document.content.

Connecting documents to layouts

Set a layout key in the document's front-matter:

---
layout: post
title: My First Post
date: 2026-01-10
---

Alternatively, configure auto-layout matching in mikser.config.js so documents are picked up by naming convention — no front-matter needed.

Running a build

npx mikser --working-folder ./my-site

Add --watch to rebuild on file changes during development. The incremental watcher only processes the documents that actually changed.

Tip: pass --clear on the first run to start from a clean output folder.
